# ヤマト × Amazon API連携 自動化プロジェクト 計画書

Amazon・Yahoo!・BASE の受注を、**ヤマト B2クラウドAPI** で送り状（伝票）自動発行し、
追跡番号を各モールへ **出荷通知** として自動連携する「全自動出荷システム」の計画です。

> 現行の「ブラウザCSV変換ツール（index.html）」は、API連携が稼働するまでの**現役の運用手段**として残します。
> また、API障害時の**フォールバック**としても有効です。

---

## 1. 実現する自動フロー（全自動）

```
[定期実行 / 数分〜数十分おき]
        │
        ▼
① 受注取得    Amazon SP-API (Orders) / Yahoo!ショッピングAPI / BASE API
        │   └─ 新規・未出荷の注文を取得
        ▼
② 伝票発行    ヤマト B2クラウドAPI（送り状発行）
        │   └─ 伝票番号（追跡番号）を取得、ラベルPDFを取得/印刷
        ▼
③ 出荷通知    Amazon SP-API (confirmShipment) / 各モールの出荷通知API
        │   └─ 運送会社=ヤマト + 追跡番号 を連携
        ▼
④ 記録        出荷済みフラグを保存（重複出荷を防止）
```

## 2. 技術スタック（推奨）

| 領域 | 推奨 | 補足 |
|---|---|---|
| 言語 | **TypeScript (Node.js)** | 各APIのSDK/型が充実 |
| 実行基盤 | **AWS Lambda** | サーバーレス・使った分だけ課金 |
| 定期実行 | **Amazon EventBridge Scheduler** | cronで自動起動 |
| 状態保存 | **DynamoDB**（または S3） | 出荷済み・冪等性キーの管理 |
| 秘密情報 | **AWS Secrets Manager / SSM Parameter Store** | APIキー・トークンを安全に保管 |
| 監視 | **CloudWatch Logs + アラート** | 失敗時にメール通知 |
| IaC | **AWS SAM / CDK** | 構成をコード管理・再現可能に |

> 低コスト重視なら **Cloudflare Workers + Cron Triggers + KV/D1** も有力（無料枠が大きい）。
> 既存の受注管理システムがある場合はそこへ組み込む選択肢もあります。

想定コスト：小規模（1日数百件まで）なら **月数百円〜千円程度**（Lambda/EventBridge/DynamoDB の無料枠＋少額）。

## 3. フェーズ計画

### Phase 0：申請・準備（最優先・あなたの手続きが必要）
- [ ] **Amazon SP-API** 開発者登録・アプリ作成・ロール申請（後述）
- [ ] **ヤマト B2クラウドAPI** 申請・契約・仕様書入手（後述）
- [ ] （任意）Yahoo!ショッピングAPI / BASE API の利用申請
- [ ] AWS アカウント作成

### Phase 1：基盤構築
- [ ] AWS 環境（Lambda/EventBridge/DynamoDB/Secrets）を IaC で構築
- [ ] 秘密情報の登録、ロギング・通知の整備

### Phase 2：受注取得（読み取りのみ・安全）
- [ ] Amazon SP-API から未出荷注文を取得 → 正規化（現行ツールの変換ルールを移植）
- [ ] 取得結果をログ/プレビューで検証（まだ伝票発行はしない）

### Phase 3：伝票発行（ヤマト）
- [ ] B2クラウドAPI で送り状発行 → 伝票番号取得
- [ ] **冪等性**（同一注文を二重発行しない仕組み）を実装
- [ ] まずは**手動承認（半自動）**で検証 → 問題なければ全自動へ

### Phase 4：出荷通知＋全自動化
- [ ] Amazon confirmShipment で「ヤマト運輸＋追跡番号」を連携
- [ ] Yahoo!・BASE の出荷通知も同様に実装
- [ ] EventBridge で定期実行 → 全自動運用開始
- [ ] 失敗時の再試行・通知・手動リカバリ手順を整備

## 4. API申請の手引き（Phase 0でやること）

### ① Amazon SP-API（Selling Partner API）
1. セラーセントラルにログイン
2. 「アプリ＆サービス」→「アプリを開発」→ **開発者登録**（会社情報・用途・データ保護方針の申請）
3. **自社アプリ（プライベートアプリ）を作成** → LWA クレデンシャル（client id / secret）取得
4. 必要な **ロール**（Orders、出荷確認 等）を申請
5. **リフレッシュトークン**を発行（自社セラーアカウントを認可）
- 使うAPI：Orders API（`getOrders`/`getOrderItems`）、`confirmShipment`（出荷通知）

### ② ヤマト B2クラウドAPI
1. ヤマトビジネスメンバーズ / 担当営業経由で **B2クラウドAPIの利用を申請・契約**
2. 審査後、**API仕様書・接続情報**を入手（※仕様は契約者のみ開示）
3. 「画面ありAPI／画面なしAPI」のどちらを使うか選定（全自動なら画面なしが基本）
- 参考：https://business.kuronekoyamato.co.jp/service/lineup/b2api/index.html

### ③（任意）Yahoo!ショッピングAPI / BASE API
- Yahoo!：Yahoo!ショッピング 商品/注文連携API（出店者向け、利用申請が必要）
- BASE：BASE Developers でアプリ登録 → OAuth → 注文取得・出荷登録API

## 5. 設計上の重要ポイント

- **冪等性（重複出荷の防止）**：注文IDをキーに「処理済み」を DynamoDB に記録。再実行されても二重に伝票発行/通知しない。
- **変換ルールの再利用**：現行 `index.html` の正規化ロジック（出荷日=お届け−2日、品名短縮、住所スペース整形、電話 +81→0、同一宛先統合）を**サーバー側に移植**して流用。
- **段階的リリース**：いきなり全自動にせず「取得のみ→半自動→全自動」と安全に移行。
- **秘密情報をコードに置かない**：トークン類は Secrets Manager 等で管理（公開リポジトリに絶対コミットしない）。
- **エラー処理**：API失敗・在庫切れ・住所不備などは自動でスキップ＋通知し、手動対応できるように。
