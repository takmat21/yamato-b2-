# API申請ガイド（Amazon SP-API / ヤマト B2クラウドAPI）

全自動出荷バックエンドを動かすには、以下2つのAPI利用申請と認証情報の取得が必要です。
申請フォームの送信はご本人のアカウント（Amazonセラー／ヤマト法人契約）でのログインと
本人確認が必要なため、**このガイドのとおりにご本人で進めてください**。取得した認証情報を
私（開発側）へ渡していただければ、実装〜通し検証まで進めます。

> 🔒 取得する認証情報は**チャットに直接貼らない**でください。漏えい対策として
> AWS Secrets Manager 等に登録するか、安全な受け渡し方法を別途決めます（最後の章参照）。

---

## A. Amazon SP-API（自社利用／プライベートアプリ）

自社のAmazon出品アカウントの注文を読む・出荷通知する用途なので、一般公開アプリではなく
**「自分のセラーアカウント専用（プライベートアプリ）」**として登録します。

### 前提
- Amazon出品アカウント（大口出品を推奨）／セラーセントラルの**管理者権限**
- 開発者プロフィールの本人確認（審査に数日かかることがあります）

### 手順
1. **開発者登録**
   セラーセントラルにログイン →「アプリストアとサービス」→「アプリを開発」（Develop Apps）
   →「開発者プロフィール」を登録。会社情報・利用目的・データ保護方針などを入力し、本人確認。
   → 承認されるまで待つ（数日のことあり）。
2. **アプリ（アプリクライアント）を登録**
   「Add new app client」から新規アプリを作成。
   - アプリ名：例 `zazoo-fulfillment`
   - API種別：**SP-API**
   - **ロール（権限）**：注文取得と出荷通知に必要な範囲を選択
     - 注文情報の閲覧（Orders）
     - 出荷確認（Shipping / Fulfillment 系）
     - ※ **お届け先の氏名・住所・電話などの個人情報(PII)** を読むには、PIIデータへの
       アクセス許可が必要で、**追加審査**が入ることがあります。実行時は Tokens API で
       **RDT（Restricted Data Token）**を取得して PII を読みます（実装側で対応します）。
   - OAuthリダイレクトURI：自己認可のみなら仮の `https://example.com/callback` でも可
3. **認証情報を取得**
   アプリの「LWA認証情報を表示」から
   - **LWA クライアントID**（`amzn1.application-oa2-client.xxxx`）
   - **LWA クライアントシークレット**
   - （アプリケーションID）
4. **自己認可してリフレッシュトークンを取得**
   アプリ一覧の「Authorize」→ 自分のセラーアカウントを認可 →
   - **リフレッシュトークン**（`Atzr|...`）
   > ⚠️ **リフレッシュトークンは一度きりの表示**のことがあります。消える前に必ず保存。

### 固定値（日本マーケットプレイス）
- マーケットプレイスID：`A1VC38T7YXB528`
- エンドポイント（極東）：`https://sellingpartnerapi-fe.amazon.com`

## B. ヤマト B2クラウドAPI

「送り状発行システムB2クラウド」をAPIから操作し、送り状（伝票番号）を自動発行します。
**法人のお客さま**が申し込めます。全自動運用には**画面なしAPI**を選びます。

### 前提
- ヤマト運輸の法人契約（B2クラウド利用中）
- **ヤマトビジネスメンバーズ（YBM）**のログインID

### 手順
1. **B2クラウドAPIの利用申し込み**
   B2クラウドAPIの案内ページ、またはYBM内の手続き、もしくは担当のセールスドライバー／
   支店経由で申し込み。
   - 公式案内：`https://business.kuronekoyamato.co.jp/service/lineup/b2api/index.html`
2. **API種別の選択**
   - **画面なしAPI**：B2クラウド画面を開かずに送り状発行（＝全自動向け。こちらを選択）
   - 画面ありAPI：取込結果を画面で確認・修正してから発行（半自動向け）
3. **取込レイアウトの選択**
   - **基本レイアウト**：ヤマト指定フォーマット（=現行ツールが出力している95列）
     → 既存の変換ロジックをそのまま流用できるので**基本レイアウトを推奨**
   - 新規レイアウト：独自フォーマット
4. **APIアクセス認証キーの確認**
   YBMにログイン →「ご利用中のサービス」→「送り状発行システムB2クラウド」→
   B2クラウドメニュー →「外部システムとの連携」→ 表示される
   **「APIアクセス認証キー」**をコピー。
5. 接続先URL・顧客コード等、申し込み後に案内される接続情報を控える。

### 問い合わせ
- 一般FAQ（法人）：`https://faq-biz.kuronekoyamato.co.jp/`
- 不明点は担当セールスドライバー／支店、またはYBMサポートへ。

## C. 申請後に私へ渡す情報（このまま設定値になります）

`backend/.env.example` を埋める形で控えてください（値そのものはチャットに貼らない）。

| 項目 | 環境変数 | 取得元 |
|---|---|---|
| LWAクライアントID | `LWA_CLIENT_ID` | Amazon アプリ |
| LWAクライアントシークレット | `LWA_CLIENT_SECRET` | Amazon アプリ |
| リフレッシュトークン | `SPAPI_REFRESH_TOKEN` | Amazon 自己認可 |
| マーケットプレイスID | `SPAPI_MARKETPLACE_ID` | `A1VC38T7YXB528` |
| エンドポイント | `SPAPI_ENDPOINT` | `https://sellingpartnerapi-fe.amazon.com` |
| ヤマトAPI接続先 | `YAMATO_API_BASE` | ヤマト案内 |
| ヤマト顧客コード | `YAMATO_CUSTOMER_CODE` | ヤマト契約 |
| APIアクセス認証キー | `YAMATO_API_KEY` | YBM「外部システムとの連携」|

---

## D. 認証情報の安全な受け渡し

1. 検証段階：`backend/.env`（gitignore済み）にローカルで設定 → `DRY_RUN=true` で動作確認。
2. 本番：**AWS Secrets Manager / SSM Parameter Store** に登録し、Lambdaから参照
   （`template.yaml` に参照例をコメントで記載済み）。
3. リフレッシュトークン・認証キーは**リポジトリへ絶対にコミットしない**。

---

## 申請の進め方まとめ
1. まず **Amazon 開発者登録**（審査待ちが発生するので最優先で着手）
2. 並行して **ヤマト B2クラウドAPI 申し込み**（画面なし／基本レイアウト）
3. 認証情報が揃ったら私に連携 → `channels/amazon.ts`・`carriers/yamatoB2.ts` を実装 →
   `DRY_RUN=true` で通し検証 → 問題なければ本番稼働
